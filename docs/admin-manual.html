System Overview

Habitat is a full-stack mobile habit tracking application. The system allows users to create habits, check in on habits, track progress, view weekly recaps, connect with friends, and manage their profile and settings.

This admin manual is intended for system administrators, developers, or maintainers who need to install, configure, run, update, troubleshoot, or manage the Habitat system.

Primary Admin Responsibilities
  • Install and run the Habitat application locally or in a hosted environment.
  • Configure environment variables for the backend, database, and mobile app.
  • Manage the PostgreSQL database and Prisma schema.
  • Run migrations and seed test data when needed.
  • Monitor server and database availability.
  • Troubleshoot login, connectivity, API, and database issues.
  • Maintain documentation and release notes for future updates.
Main System Components
  • Mobile App: React Native application built with Expo.
  • Backend Server: Node.js and Express server that handles application requests.
  • Database: PostgreSQL database managed through Prisma ORM.
  • Authentication: User login and protected routes using token-based authentication.
  • AI Features: Habit suggestions and Wrapped summaries where enabled.

System Architecture

Habitat uses a client-server architecture. The mobile app sends requests to the backend server. The backend validates requests, applies business logic, and reads from or writes to the database.

Architecture diagram: View Habitat Architecture Diagram

Architecture Flow
  1. The user interacts with the Habitat mobile app.
  2. The mobile app sends a request to the backend API.
  3. The backend checks authentication when required.
  4. The backend processes the request using routes, controllers, and services.
  5. Prisma communicates with the PostgreSQL database.
  6. The backend returns a response to the mobile app.
  7. The app updates the screen with the newest data.
Technology Stack
  • Mobile: React Native, Expo, TypeScript
  • Backend: Node.js, Express
  • Database: PostgreSQL
  • ORM: Prisma
  • Authentication: JWT-based authentication
  • Testing: Jest, Supertest
  • Builds: Expo Application Services

Installation and Setup

This section explains how to prepare a machine to run Habitat locally. Administrators should confirm that all required tools are installed before starting the app.

Required Software
  • Node.js v18 or higher
  • npm
  • Git
  • PostgreSQL access
  • Expo CLI or Expo tooling through npm
  • Android Studio emulator or iOS simulator

Verify common tools with:

node -v
npm -v
git --version
Clone the Repository

Clone the Habitat repository and move into the project folder.

git clone https://github.com/sanikadeshmukh3/Habitat.git
cd Habitat
Run the Setup Script

The setup script is intended to simplify local startup. It should install dependencies and start the required development services.

./setup.sh

Once Expo starts, press a for Android or i for iOS when prompted.

Manual Backend Startup

If the setup script is not used, the backend can be started manually.

cd backend
npm install
npx prisma generate
npm run dev
Manual Mobile Startup

Start the mobile app separately from the backend.

cd mobile
npm install
npx expo start

Make sure the backend is running before testing login, habits, calendar, Wrapped, or friends.

System Startup Flow
  1. Start the backend server
  2. Confirm database connection is successful
  3. Start the mobile application
  4. Verify API connectivity
  5. Log in with a test user
  6. Confirm dashboard loads correctly

Configuration

Habitat uses environment variables to keep configuration separate from source code. Administrators should verify these values before running or deploying the app.

Backend Environment Variables

Typical backend variables include:

DATABASE_URL="postgresql://username:password@host:port/database"
JWT_SECRET="your-secret-key"
PORT=3000
  • DATABASE_URL connects the backend to PostgreSQL.
  • JWT_SECRET is required for login/session authentication.
  • PORT controls where the backend server runs.
Mobile Environment Variables

When the app is in production, this variable should point to the live backend API endpoint that is deployed on render.

EXPO_PUBLIC_API_URL="https://habitat-1.onrender.com"

Deployment Note: In production, the backend is hosted remotely. Ensure the mobile application points to the correct deployed API URL instead of local server.

When the app is in development, this variable should point to the local backend API endpoint.

EXPO_PUBLIC_API_URL="http://localhost:3000"

Android emulator note: Android emulators often require http://10.0.2.2:3000 instead of http://localhost:3000.

Configuration Checklist
  • Confirm backend environment variables are present.
  • Confirm the database URL points to the correct database.
  • Confirm JWT secret is set before testing login.
  • Confirm the mobile app points to the correct backend URL.
  • Restart the backend after changing environment variables.
  • Restart Expo after changing mobile environment variables.

Database Setup and Management

Habitat uses PostgreSQL to store user accounts, habits, check-ins, friend data, settings, and progress-related information.

Prisma is used as the Object-Relational Mapping (ORM) layer. It defines the database schema, generates type-safe queries, and manages migrations between schema versions.

Important Data Stored
  • Users: account information, email, password hash, timezone, profile data
  • Habits: habit name, category, frequency, visibility, active/dormant status
  • Check-ins: completion status, date, notes, difficulty rating
  • Friends: friend requests and accepted friend relationships
  • Settings: user preferences such as theme and notifications
  • Wrapped data: recap-related data or cached summaries where applicable
Generate Prisma Client

Run this after changing the Prisma schema or after installing dependencies.

cd backend
npx prisma generate
Run Database Migrations

Migrations apply schema changes to the database.

cd backend
npx prisma migrate dev

Run migrations carefully. Schema changes can affect existing data if columns are renamed, removed, or changed to required fields.

Seed the Database

Seeding creates sample users, habits, check-ins, and other test data.

cd backend
npx prisma db seed

Use seeded data for testing login, dashboard display, calendar history, Wrapped, and friends.

Open Prisma Studio

Prisma Studio provides a visual interface for inspecting and editing database records.

cd backend
npx prisma studio

Warning: Manual edits can affect streaks, completion rates, login behavior, and recap generation. Edit production data carefully.

Recommended Data Management Practices
  • Back up important data before major schema changes.
  • Use migrations rather than manually changing database structure.
  • Confirm seeded users are only used for testing.
  • Avoid editing password fields manually unless absolutely necessary.
  • Check user IDs and habit IDs carefully before editing related records.
  • Document any manual database changes for future maintainers.

Maintenance and Updates

Regular maintenance helps keep Habitat stable and easier to support.

Routine Maintenance Checklist
  • Confirm the backend server starts successfully.
  • Confirm the mobile app can reach the backend.
  • Confirm login works with a valid test user.
  • Confirm habits load on the dashboard.
  • Confirm calendar check-ins save correctly.
  • Confirm Wrapped loads when enough data exists.
  • Check for dependency or package errors after updates.
Updating Code

Pull the latest project changes:

git pull origin main

Reinstall dependencies if package files changed:

npm install
Updating the Database

After schema changes, run:

cd backend
npx prisma migrate dev
npx prisma generate

If sample data is needed after a reset or new environment setup, run the seed command.

Restarting Services

Restart the backend after configuration, dependency, or database changes. Restart Expo after changing mobile environment variables or package versions.

Release Maintenance
  • Update release notes when features or known issues change.
  • Verify binary download links before submission or demo.
  • Confirm test credentials are valid.
  • Confirm the hosted backend and database services are active before demonstration.
  • Keep user and admin manuals updated with current functionality.

Testing

Testing helps verify that important parts of Habitat continue to work after changes.

Run Backend Tests
cd backend
npm test

Tests should be run after major backend changes, authentication updates, database schema changes, or AI summary updates.

Run Frontend Tests
cd mobile
npm test

Tests should be run after major frontend changes, navigation updates, or UI changes that may affect user flows.

Test Areas
  • User registration and login
  • Protected routes
  • Habit creation, editing, and deletion
  • Habit check-ins
  • Streak tracking
  • Calendar data rendering
  • Habit stacking
  • Wrapped logic
  • Friend requests and friend lists
  • AI integration for creating habits and generating summaries
  • Profile and badge management
Manual Smoke Test
  1. Start the backend.
  2. Start the mobile app.
  3. Log in with a test user.
  4. Create a habit.
  5. Complete a check-in.
  6. Verify the dashboard updates.
  7. Open the calendar and confirm the check-in appears.
  8. Open the habit detail screen and confirm progress displays.
  9. Open Wrapped and confirm the recap screen loads.

Security and Access Notes

Environment Secrets
  • Do not commit real environment files to the public repository.
  • Keep database credentials private.
  • Keep JWT secrets private.
  • Rotate exposed secrets immediately if accidentally shared.
User Data
  • User notes and difficulty ratings should be treated as private.
  • Friend profiles should only show public habits.
  • Manual database edits should be limited to authorized maintainers.
  • Test data should not be confused with real user data.

Troubleshooting

Backend server will not start

Run npm install, confirm Node.js is version 18 or higher, and check whether another process is already using the backend port.

Port 3000 is already in use

Run lsof -i :3000 to find the process, then stop it or change the port.

Database connection fails

Confirm the database is running and that DATABASE_URL points to the correct database.

Prisma client errors occur

Run npx prisma generate from the backend folder. If the schema changed, also run migrations.

Migration fails

Check whether the database already contains conflicting schema changes. Review the migration error message before resetting or editing the database.

Login does not work

Confirm the backend is running, the database contains the user, the password is correct, and JWT_SECRET is configured.

Mobile app cannot connect to backend

Confirm EXPO_PUBLIC_API_URL is correct. For Android emulator, try http://10.0.2.2:3000.

Data does not update in the app

Refresh the screen, restart the app, and confirm the backend returned the expected response. Also check whether cached data needs to be invalidated or refreshed.

Wrapped does not show a full recap

Confirm the user has enough habit check-in activity for the selected week. Wrapped depends on recorded habit activity, notes, and ratings.

AI features fail

Check whether required AI configuration is present and whether the backend route is returning an error. Review backend logs for the exact failure.

Expo app opens but features fail

This usually means the mobile app is running but cannot reach the backend. Confirm the API URL and backend status.

Build link expired

Generate a new build using Expo Application Services and update the documentation link.

Checking Logs
  • Backend logs appear in the terminal running the server.
  • Look for API errors, authentication failures, or database errors.
  • Expo logs appear in the terminal running the mobile app.
  • Use logs to trace request failures or unexpected behavior.
